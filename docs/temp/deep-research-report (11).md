# Текущее состояние системы

- **POS Edge (кассовый backend).** Реализован полный базовый кассовый сценарий: создание заказа, выпуск precheck, захват платежей и формирование итогового чека (Order → Precheck → Payment → Check)【4†L12-L15】【1†L8-L11】. Реализованы скидки/надбавки и налогообложение по заданному pipeline (отдельные объекты *discount*/*surcharge* с `application_index`, Tax Always Last, детерминированное округление)【4†L12-L15】【54†L288-L296】. Локальная сессия кассира и личная смена под управлением (открытие/закрытие, PIN-аутентификация) работают; есть API для смены кассы и регистрации событий денежного ящика. Отмена precheck требует override менеджера (PIN/роль проверяются) и возвращает заказ в “open”【4†L12-L15】【12†L244-L252】. Поддерживаются частичная оплата precheck и создание итогового чека только после полной оплаты【4†L13-L15】. Операции отмены и возврата записываются в append-only ledger (`financial_operations`/`financial_operation_items`) без изменения финализированного чека【4†L14-L16】. Подсистема «выходящая синхронизация» (outbox) построена с сохранением `pending/processing/sent/failed` и retry-safe логикой (idempotent resend, экспоненциальный бэкофф, автоматический recovery)【4†L25-L28】.  

- **Локальные read-модели и настройки.** На Edge хранятся справочники залов/столов, меню (items, категории, модификаторы), сотрудники/роли, политики ценообразования и т. д., получаемые из Cloud по публикуемым пакетам. Существуют HTTP-эндпоинты `/halls`, `/tables`, `/catalog/items`, `/menu/items` и т. п. (см. [`pos-backend` router](https://github.com/serty2005/mh-pos/blob/master/pos-backend/internal/pos/api/router.go)【15†L49-L57】【15†L52-L60】). В коде видно, что на Edge-стороне чтение этих данных реализовано (в таблицах SQLite есть `halls`, `tables`, `menu_items`, `modifier_groups` и т. д., см. схему【26†L17-L26】【27†L30-L36】). Таким образом персонал может видеть меню, залы и т.д. офлайн.

- **POS UI (касовый клиент).** Интерфейс на Vue/Quasar покрывает основные cashier-функции. По README: после регистрации устройства и входа по PIN кассиру доступны открытие смен, выбор зала/стола, создание заказа, добавление позиций (включая услуги и модификаторы), изменение количества, вынесение пречека, оплата и печать чека, отмены и возвраты через ledger【69†L112-L119】【69†L120-L127】. Таким образом cashier UI фактически поддерживает полный цикл продажи. **Однако** явного разделения “режим кассира” vs “режим официанта” в UI нет – текущее POS-приложение ориентировано именно на стационарную кассу. Особое мобильное “waiter mode” отсутствует【69†L112-L119】【69†L100-L103】. 

- **Cloud Backend (master-сервер).** Реализован приём и хранение операций с Edge: есть HTTP‐приёмник `POST /api/v1/sync/edge-events` (и батч-версия), сохраняющий сырые события (`cloud_operational_events`) и заполняющий проекции (финансы смен, статистика)【34†L143-L152】【34†L166-L174】. Имеются CRUD‑API для всех основных справочников: рестораны, роли, сотрудники (с PIN-хэшами), залы/столы, каталоги и пункты меню, модификаторы, политики ценообразования и пр. (см. [router Cloud master-data](https://github.com/serty2005/mh-pos/blob/master/cloud-backend/internal/masterdata/api/router.go)【35†L21-L30】【35†L87-L96】). Реализован workflow публикации master-data: при вызове `POST /restaurants/{id}/master-data/publish` формируются versioned пакеты (`restaurants`, `staff`, `floor`, `catalog`, `menu`, `pricing_policy`), которые потом Edge получает при подключении【30†L65-L74】. Реализована схема развертывания устройств (pairing/provisioning) через Cloud-согласование и License-коды【30†L17-L20】. Есть очереди `inventory_event_queue`, `stock_documents`, `stock_ledger` в базе – то есть подготовлено место для учёта складских операций【30†L53-L55】; в README говорится, что “Cloud Inventory Worker” уже пишет нормализованные документы в `stock_ledger` из событий инвентаризации【67†L1-L10】. Фактическая бизнес‐логика расчёта себестоимости и обработки рецептов пока не реализована. 

- **Cloud UI (админский веб-приложение).** Фронтенд на Vue/Quasar предоставляет **относительно схематичный админ‑интерфейс**. Подтверждены экраны/CRUD для ресторанов, ролей, сотрудников, каталога, модификаторов, политик ценообразования, залов/столов, пунктов меню и процесса публикации master-data【72†L40-L48】. То есть сделан базовый “админ-панель”: формы и таблицы для всех справочников. Дизайн пока ближе к техническому CRUD (вроде встроенного шаблона Quasar) – здесь отсутствуют продуманные UX-паттерны (onboarding, визуальные конструкторы меню/залов, продвинутые дашборды и пр.). Cloud UI отражает именно текущий API (см. [cloud-ui README](https://github.com/serty2005/mh-pos/blob/master/cloud-ui/README.md)【72†L8-L12】): то есть реализованы только те операции, которые точно есть в backend (CRUD и публикация).

- **Неполные/не реализованные зоны (foundation only).** В коде **есть таблицы** для рецептов (`recipe_versions`, `recipe_lines`) и стоп-листов (`stop_lists`), но самих API и бизнес-логики по ним в runtime нет【20†L60-L65】【60†L54-L56】. То же касается сложной работы с рецептурами и складом: таблицы `stop_lists`, `inventory_event_queue` существуют, но механика блокировки продаж или учёта потребления компонентов не работает (такие процессы указаны лишь в документации). **KDS/кухонный поток** никак не реализован – ни веб-сокетов, ни экранов кухни, ни статуса блюд нет (интерфейсы POS-backend и Cloud-backend не содержат сегментов “production queue” или “item served” и т.п.). **BI-дэшборды и отчёты** на Cloud построены только по журнальным данным: реализован проекционный столбец для финансовых операций, но готовые отчёты или визуализация отсутствуют. Кроме того, правила офлайн-синхронизации (двунаправленный sync stop-листа, конфликтные ситуации) на текущем уровне кода не покрыты; фактически единственный owner стоп-листа в схеме – Cloud (поля `cloud_version` в таблице), Edge лишь читает snapshot из публикаций (и даже не предусмотрено клиентского CRUD для стоп-листа). 

- **Консистентность и защита.** Код содержит механизмы проверки прав и целостности: сессии и PIN-авторизация, проверка прав в сервисах (`EnsureOperatorSession`), защита от дублирования команд (проверка `command_id`)【10†L93-L100】【54†L298-L307】. Избыточные сценарии, которые могли бы спровоцировать рассинхронизацию, тщательно обрабатываются: например, применено контрактное разделение *node_device_id* и *client_device_id* при авторизации【69†L132-L140】【26†L10-L14】, есть idempotent retry у sync-outbox【4†L25-L28】. Миграции устроены строго: до первого запуска БД создаётся заново из baseline, откатные миграции не поддерживаются. Из README: если checksum миграций не совпадает, сервис не стартует【4†L30-L33】【30†L53-L55】. Это обеспечивает safety, но также означает, что схемы (и соответственно код) должны очень аккуратно обновляться, чтобы не “сломать” живую БД. 

# Зона пилотного запуска

На **пилотном запуске ресторана** нам нужен полностью функциональный кассирский и менеджерский опыт, а также базовый кухонный. Минимальный scope по ролям:

- **Кассир (POS Edge + POS UI):** открытие смен (личной и кассовой), выбор зала/стола, ввод заказа (пункты меню и услуги), выбор модификаторов, изменение количества/аннулирование позиций, выпуск precheck, оплата (наличный/карточный через precheck), печать чека, отмена/возврат чека (через ledger), закрытие смены. Это **почти готово**: все этапы реализованы в бекенде и UI【4†L12-L15】【69†L112-L119】. Для пилота важно проверить готовность offline-сценариев (Edge без связи), но так как POS-UI работает по локальному JSON API (и получить полные данные меню/залов она уже умеет через синхронизацию), базовая непрерывность покрыта.

- **Менеджер (Cloud Backend + Cloud UI):** администрирование справочников (ресторан, сотрудники/роли, залы/столы, каталоги/меню, модификаторы, политики ценообразования, рецепты если есть), публикация мастер-данных в Edge, просмотр операций (отчёты по продажам, корректностям). В **текущем состоянии** Cloud Backend поддерживает CRUD и публикацию master-data для каталогов и меню【35†L79-L88】【72†L40-L48】. Cloud UI позволяет создавать/редактировать рестораны, персонал, меню и т. п.【72†L40-L48】, но пока без встроенных аналитик или дашбордов. Для пилота необходимо закончить (или хотя бы интерфейсно обеспечить) такие менеджерские потоки как заполнение меню/модификаторов, установку правил цены и налогов, настройку половины интерфейса стоп-листа. Сейчас редактор меню/каталога и модификаторов на Cloud UI есть, но **визуальный конструктор зала (drag&drop)** и прочие удобства отсутствуют; это можно отложить на пост-пилот, если в пилоте достаточно ручной настройки в существующих формах.

- **Кухня (KDS, кухонный поток):** хотя KDS (Kitchen Display System) критичен для полной системы, в пилоте можно начать с **очень простого решения** – например, вручную передавать через чек статус “готово” или “отменено”. В коде нет реального KDS (нет WebSocket, очереди готовки, смены статусов блюд), поэтому пока **пилотный поток кухни минимально**: метки «готовки» могут ставиться через чек в POS, а на Cloud срабатывать соответствующие события (например, `ItemServed`, `ProductionCompleted`). Гораздо важнее обеспечить **правильное списание ингредиентов** (Stock Consumption). Но так как движок списания ещё не готов, на пилоте можно отслеживать списания вручную (через стоп-лист). 

- **Стоп-лист (компенсации и остатки):** стоп-лист – критичный пункт. Для пилота нужно **блокировать продажу ингредиентов и блюд, исчерпавших запас**. В коде на Edge есть таблица `stop_lists`【60†L54-L57】, но нет бизнес-логики проверки при добавлении позиции. То есть на данный момент **при продаже ничего не блокируется**. Для пилота это необходимо реализовать: при попытке добавить позицию система должна сверяться с `stop_lists` (и с развёрнутой рецептурой) и выдавать ошибку, если ресурс исчерпан. Анализ кода показал, что это **не сделано** (нет соответствующего hook в `addOrderLine`), поэтому требуется доработка этой логики.

- **Синхронизация стоп-листа (Edge ↔ Cloud):** на Edge есть только чтение из Cloud-публикаций, а Cloud не предоставляет API для стоп-листа. Нужен механизм, кто “владеет” стоп-листом. Скорее всего, авторитетным источником будет Cloud (через облачное приложение, менеджер сам отмечает блюда как “запрет” или “разрешено с остатком”). Тогда надо добавить API Cloud для стоп-листа и синхронизировать это на Edge. Иначе, если Edge имеет офлайн-экземпляр, нужен ещё двусторонний merge (списания можно зеркалить).

- **Inventory / склад:** текущая реализация Cloud смежных событий достаточна для простой отчётности, но **полного учёта** (реального списания по завершении чека) нет. Для пилота можно сделать упрощённый вариант: при создании чека фиксировать расход (консумированное количество *normalized*). Более сложные задачи (рецептурное списание, возвраты в склад) пока отложить.

# Основные блокеры

- **Стоп-лист (Критично).** Нет реализации блокировки продажи “запрещённых” ингредиентов/блюд. Если не доделать, на пилоте можно случайно продать то, чего нет. Несмотря на наличие таблицы, логика отсутствует в коде – это становится **критическим блокером**.  

- **KDS/кухонный поток (Критично/Высоко).** Полноценная система статусов блюд отсутствует. Без хотя бы минимальной KDS-функции невозможно обрабатывать кухню. Если пилот предполагает кухню, нужно срочно заложить простую логику (пусть даже через чек: “заказчик съел блюдо” -> `ItemServed`/`ProductionCompleted` событие). Сам экран KDS может быть примитивным.  

- **Waiter UI (Высоко).** Изначально сказано, что пилот – «mobile-first waiter runtime» и отдельный режим официанта. Но текущий POS UI не поддерживает ничего подобного【69†L112-L119】【69†L100-L103】. Для пилота нужен хотя бы упрощённый мобильный интерфейс официанта (например, веб-страница, где официант видит свой активный стол/заказы). Разделение ролей в UI пока не предусмотрено, поэтому придётся расширять фронтенд.  

- **UI / UX менеджмента (Средне).** Cloud UI пока больше похож на *”json-ориентированный CRUD”*. Отсутствуют удобные паттерны: визуальный редактор зала, пошаговый онбординг, bulk-редактирование меню, review-процессы публикации и т.п. Это не стопер для базового пилота (все операции можно делать через существующие формы), но с точки зрения product-grade интерфейса – большой долг. Стоит минимально улучшить комфорт (напр. добавить фильтры, подсказки, логи ошибок синхронизации).  

- **Синхронизация и конфликт-менеджмент (Средне).** При двунаправленном обмене (например, если Edge должен уметь вносить коррективы или обрабатывать сбои) нужно проверять, не повиснут ли “подтверждения” в ожидании. Сейчас master-data инволюционирует строго из Cloud → Edge (Edge не делает PUT того же ресурса). Это упрощает ситуацию, но добавляет боль: любое ручное изменение на Edge потеряется при синке. Нужно чётко определить ownership и, возможно, добавить проверки (например, предупреждения админа, если какой-то объект имеет `cloud_version` > `last_synced_at`).  

- **Техническая устойчивость (Низко/Средне).** Большинство путей защищено, но стоит проверить: что если операции архивации/очистки данных выполнены некорректно или промигрировали? Отладочные команды (retention, archive) уже есть, но finalize-операции (`VACUUM`, ресинк) не реализованы【1†L21-L23】. Для пилота убедиться, что хранения хватает, и что нет незакрытых миграций.

# Рекомендуемый порядок доработок

1. **Стоп-лист (добавление проверки на Edge).** Сначала реализовать бизнес-логику запрета продаж: при `AddOrderLine` делать lookup по `stop_lists` (и по расширенной рецептуре). Это критично: позволит вообще остановить продажу “на стопе”. Без этого пилот теряет смысл.  

2. **Минимальная KDS-логика.** Организовать события «поступление блюда» и «приготовление» через чек или другой механизм: например, автоматически считать `ProductionCompleted` для каждой позиции при закрытии чека. Это фундамент для учета на кухне. Затем – простейший интерфейс KDS (даже текстовый список заказов).  

3. **Разработка waiter UI.** Параллельно с KDS начать мобильный интерфейс официанта: отдельный “режим” POS UI для официанта (role-aware навигация, упрощённая форма заказа/оплаты). Это позволит пилоту обслуживать столы мобильными устройствами.  

4. **Интеграция стоп-листа в Cloud.** После того как Edge сможет читать стоп-лист, нужно решить, как его наполнять. Либо дать менеджеру Cloud UI возможность редактировать стоп-лист (API Cloud и screens), либо предусмотреть синхронизацию с внешней системой (если такая есть). Без этого Edge не узнает, какие позиции блокировать.  

5. **Инвентаризация/склад (выделенные задачи).** Если пилот требует учёта ингредиентов, продумать схему: возможно, на первом этапе фиксировать только manual-инвентори (через Cloud) и давать read-only отчёты. Сложные функции (авторецептура, автоматический списание) оставить далее.  

6. **Улучшение Cloud UI.** После запуска базовых функций постепенно расширять UX: добавить search/filter в списках, визуальные редакторы (меню, залы), отчётность (оперативные и сменные отчёты). Это можно делать по приоритету по мере обратной связи пилота.  

7. **Постройка отчётов и дашбордов.** В Cloud Backend постепенно выводить нужные проекции (например, total sales по дням, top-10 блюд, остатки), и визуализировать в Cloud UI.  

Каждый шаг логичен по зависимостям: нельзя полноценно работать на кухне без стоп-листа, нельзя официантам запускать мобильный order без waiter UI. При этом критичные модули (стоп-лист, KDS) надо решить до пилота, а второстепенные (анализ данных, красивая визуализация) – после.

# UX/UI-аудит

- **Cashier UI (pos-ui):** На текущий момент интерфейс **кассы** решает задачу, но обладает ограниченным UX. Положительно: используется понятная секция услуг, модификаторов и “чек” просматривается по snapshot【69†L112-L119】. Недостаточно: много ручного ввода (нет подсказок к столам, нет контекстных меню). Нужно адаптировать под мобильное устройство (текущий дизайн — скорее десктопный). Целевой “cashier” экран стоит разделить от “lock/login”/“shift” экранов – сейчас это одно приложение без явных ролей. Например, добавить большую кнопку “Открыть смену”, хранить таблицу сгенерированных чеков, улучшить контроль переходов между экранами.  

- **Waiter UI:** Отсутствует. Требуется отдельный lightweight интерфейс для официанта. Идея: показывать офицанту только свои столы/заказы, упрощать ввод (сканирование QR стола, автодобавление блюда в чек). Так как ничего готового нет, минимум – выделить кнопку “Официант” на login-screen, которая переключает на мобильно-адаптированный layout (крупные кнопки, списки блюд без услуг и подробностей). Важный UX-паттерн: drag-and-drop стол (или карта зала), быстрый заказ/оплата на ходу. Для пилота достаточно «мобильной» стилизации текущих форм и роли офицанта – полная переделка можно отложить.  

- **Cloud Manager UI:** Это именно “enterprise admin” интерфейс. Сильные стороны: есть все необходимые формы для CRUD основных сущностей【72†L40-L48】. Но наблюдается технологическая фокусировка: список сущностей выводится в таблицах с редактированием по форме, мало помощников. Недостатки: нет онбординга (первый запуск ресторана сводится к заполнению множества форм без пошаговых подсказок), нет визуального построения меню или зала (валидная UX-фича – перетаскивать категории/блюда между разделами), нет панелей мониторинга (сейчас нужно смотреть отчёты, а дашбордов нет). На пилоте это повлияет на скорость внедрения, но пока критично обеспечить работоспособность через базовые CRUD: это есть. В дальнейшем следует: сформировать “wizard” первого запуска (основное меню, первая роль, первая категория), добавить предпросмотр публикаций (что Edge увидит), улучшить поиск/фильтры в списках.  

- **UX/Conformity с рынком:** По сравнению с современными RMS/POS (iiko, Toast, Clover и т.д.), система отстаёт в опыте пользователя. Там часто есть:
  - пошаговые setup и тренинги;
  - графические редакторы меню и планов зала;
  - встроенные KPI-доски (например, свежие продажи, задержанные заказы);
  - гибкие workflows (approve-флоу для изменений в меню/ценообразовании);
  - уведомления о синхронизации ошибок и статусах устройств. 
  Здесь многое придется либо руками покрывать, либо сильно упростить. Главное – не допустить «человеческого факапа»: например, добавить предупреждение при публикации master-data и отслеживание сбоев.

# Технический долг и риски

- **Архитектурный дрейф.** Документация описывает идейную модель (SPECv1.3, ROADMAP), но код иногда от неё отстаёт. Например, документация упоминает бюджетовые подсчёты и inventory, а в коде эти модули ещё в плане. При расхождении источник – код. Нужно внимательно сверить: если спецификация говорит об интеграции с PSR или фискализацией, но в репозитории нет кода, пометить как «вне спектра пилота».  

- **Дублирование и нарушения границ.** Код разделён по DDD, но есть моменты “leakage”: например, UI-клиент иногда передаёт `device_id` или `session_id` в явном виде, что могло бы быть скрыто. В ряде сервисов (см. CancelPrecheck) параметры передаются командой, вместо того чтобы читаться из текущей сессии. Надо убедиться, что авторизация идёт через `CommandMeta`, а не через параметры URL, во избежание рассогласования.  

- **Hazards в синхронизации.** Обозначена retry-safe логика outbox【4†L25-L28】, но что если события приходят не по порядку? Пример: если несколько изменений справочников подряд, надо гарантию, что применяются версии. Сейчас приёмник Event hub просто записывает в очередь – возможно, имеет встроенную сортировку по `sequence_no`. Стоит проверить, что `last_cloud_version` в `cloud_master_sync_state` обновляется правильно, и что дубли не крашнут систему.  

- **Хазарды ценообразования.** Система зависит от детерминированности расчёта (integer math), что хорошо. Потенциальный риск – отсутствие явных чек-поинтов после формирования счёта: скидки и надбавки просто суммируются. Нужно убедиться, что округление всегда соответствует ожиданиям (есть тесты schema, но лучше smoke-тесты по вычислению цен).  

- **Миграции и данные.** Как указано в README, без первого production-базы никаких data-preserving upgrades нет【4†L30-L32】. Это упрощает пилот (все начинают с чистой), но добавляет долг: когда придёт миграция, команда должна разработать процедуру «скрипта миграции» и убедиться, что все старые записи распознаются. Особенно опасно, если будут изменения в части `sync` (новые streams) – нужно ручное вмешательство админа.  

- **UI-авторитетности логики.** Фронтенд не должен принимать решения вместо бекенда – это сделано верно (UI клиент лишь вызывает API, а подсчёты делает backend【69†L112-L119】). Следует продолжать в том же духе: валидации и бизнес-логика должны оставаться на сервере.

# Рекомендации по Pilot Release

- **Реалистичный объём пилота:** Выпустить должен **функционирующий кассовый узел** с управлением основными процессами и доступом для менеджера. То есть кассир может обслуживать столы и проводить продажи без связи с сетью, а менеджер может настраивать меню/каталоги и отслеживать финалы смен. Система *должна быть устойчива к сбоям* (offline-сохраняемость транзакций, простая роль учёта остатков через стоп-лист).  

- **Что нельзя допускать:** Нельзя выпускать пилот без механизма стоп-листа (иначе возможны неразрешённые продажи) и без безопасности (всех операций в кассе без проверки PIN менеджера или прав). Также критично, чтобы отмены/возвраты работали «правильно» – иначе расход/приход будут рассинхронизированы. Сложная интеграция с фискальными терминалами или PSP можно отложить.  

- **Приоритетные доработки перед пилотом:**  
  - **Запрет продаж по стоп-листу** – завершить прежде всего.  
  - **Рабочий поток KDS** – хоть в простом виде (событие served) должен быть перед пилотом, иначе на кухне будет хаос.  
  - **Улучшения UX кассы** (включая минимальный waiter mode), чтобы официанты могли реально использовать систему.  
  - **Синхронизация и отладка** – протестировать сценарии отработки без связи, заполнить missing locks или retries.  
  - Остальное (отчёты, bulk-редактирование, «красивости» UI) можно делать после.  

- **После пилота можно добавить:** нюансы складской книги (финансовые возвращения по остаткам), финнализацию бизнес-дня, интеграцию с внешними сервисами (платежи, фискализация, ClickHouse-аналитика) и т.д.  

В итоге пилот *должен быть* рабочим, логичным и безопасным – а не заточенным под идеальную архитектуру. Важно объяснить пилоту его ограничения (например, “все задержки продаж будут обрабатываться в Cloud UI после возвращения онлайн”), но обеспечить, чтобы бизнес-процессы работали от начала до конца. 

**Источники:** анализ кода репозитория показал текущее покрытие функций (см. например реализацию основных сервисов и маршрутов【4†L12-L15】【15†L61-L70】【69†L112-L119】) и выявил пробелы (отсутствие KDS и офицантского UI, наличие только “основы” для стоп-листа/рецептов【20†L60-L65】【69†L100-L103】). Эти данные легли в основу рекомендаций. Все выводы основаны на коде и схемах проекта (README и тесты)【4†L12-L15】【30†L53-L55】【72†L40-L48】.
