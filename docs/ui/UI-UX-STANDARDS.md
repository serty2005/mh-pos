# UI/UX Standards

## Назначение документа

Этот документ фиксирует базовые правила проектирования UI/UX для MyHoreca POS/RMS.

Важно: проект не делит UI на “Android UI” и “desktop UI” как разные дизайн-системы. Основной интерфейс делается как единая UI-система, но с разными режимами плотности и сценариями использования:

- **POS Front** — быстрый операционный интерфейс для кассиров/официантов, преимущественно на Android-планшетах и touch/touchpad устройствах;
- **BackOffice** — административный интерфейс для работы с данными, таблицами, формами, фильтрами и настройками.

Android в проекте рассматривается прежде всего как целевой класс устройств для POS Front: планшеты, touch-экраны, ограниченный viewport, работа стоя/на кассе/в зале. Это не означает, что весь продукт должен копировать Android-native UI. Платформенные Android-рекомендации учитываются только там, где они полезны для touch, размеров, читаемости и доступности.

## 1. Базовая философия UI

Проект использует два разных режима интерфейсной плотности.

### 1.1 POS Front: разуплотненный операционный UI

POS Front предназначен для скорости, низкой усталости и минимизации ошибок.

Главные свойства:

- крупный и широкий layout;
- touch-first управление;
- минимум мелких элементов;
- крупные кнопки и зоны нажатия;
- крупные иконки;
- хорошо читаемые подписи;
- минимум визуального шума;
- видимый текущий контекст: стол, заказ, пречек, сумма, смена;
- отсутствие технических деталей в рабочем интерфейсе;
- модальные потоки поверх текущего контекста;
- быстрые повторяемые действия без лишней навигации.

POS Front не должен выглядеть как admin table UI.

### 1.2 BackOffice: уплотненный data-management UI

BackOffice предназначен для управления данными.

Главные свойства:

- более плотный layout;
- таблицы, формы, фильтры, справочники;
- keyboard/mouse efficiency;
- массовые операции там, где они безопасны;
- больше информации на экране;
- compact controls;
- понятная навигация по настройкам и справочникам;
- возможность точной работы с данными.

BackOffice может быть плотнее POS Front, но не должен становиться нечитаемым или хаотичным.

## 2. Главное различие Front и BackOffice

| Характеристика | POS Front | BackOffice |
|----------------|-----------|------------|
| Основной пользователь | кассир, официант, оператор | администратор, менеджер, бухгалтер, владелец |
| Основной сценарий | быстрые операции | работа с данными |
| Устройство | Android-планшет, touch-screen, POS terminal | desktop/laptop/tablet |
| Плотность | разуплотненная | уплотненная |
| Размер кнопок | крупные | средние/компактные |
| Навигация | минимальная, контекстная | разделы, таблицы, фильтры |
| Главная цель | скорость и безошибочность | точность и полнота управления |
| Scroll | только внутренние области | допустим page-level scroll, если это удобно для data UI |
| Модалки | блокирующие, стековые | блокирующие или side panels, по сценарию |
| Текст | короткий, обрезка `...` | допускаются длинные значения, таблицы, tooltips |

## 3. POS Front viewport rule

Главное окно POS Front всегда должно влезать в размер экрана.

Обязательные правила:

- главный POS shell занимает ровно `100vw × 100vh`;
- у главного окна POS Front не должно быть document-level scrollbar;
- нельзя проектировать POS Front как страницу, которую пользователь прокручивает целиком;
- bottom navigation, action rail, текущий заказ и ключевые действия должны оставаться в пределах экрана;
- если контент не помещается, скроллится внутренняя область, а не весь shell;
- внутренний scroll допустим для menu grid, списка строк заказа, activity list, списка платежей, modal body, отчетных таблиц;
- все scroll areas должны быть touch-friendly.

Рекомендуемый инвариант:

```css
html,
body,
#app {
  width: 100vw;
  height: 100vh;
  overflow: hidden;
}

.pos-shell {
  width: 100vw;
  height: 100vh;
  overflow: hidden;
  display: flex;
  flex-direction: column;
}

.pos-main {
  min-height: 0;
  flex: 1;
  overflow: hidden;
}

.pos-scroll-area {
  min-height: 0;
  overflow: auto;
}
```

## 4. POS Front density rules

POS Front должен быть визуально свободным и неутомляющим.

Требования:

- не уплотнять экран ради отображения максимального числа сущностей;
- лучше показать меньше элементов, но крупнее и понятнее;
- меню блюд должно работать плитками, а не плотной таблицей;
- список заказа должен быть читаемым на расстоянии вытянутой руки;
- основные действия должны быть доступны без точного попадания мышью;
- destructive actions должны быть отделены визуально и требовать подтверждения;
- длинные значения должны обрезаться через `...`, а не ломать layout;
- экран не должен требовать постоянного чтения мелкого текста.

## 5. BackOffice density rules

BackOffice может и должен быть более плотным, потому что это интерфейс управления данными.

Требования:

- таблицы допускают compact rows;
- фильтры могут быть плотнее, чем POS controls;
- page-level scroll допустим, если форма/таблица объективно длинная;
- keyboard navigation и mouse interaction важнее, чем touch-first;
- bulk actions допустимы только с защитой от ошибок;
- сложные формы должны быть разбиты на логические секции;
- длинные значения могут показываться в таблицах, tooltips, side panels или detail pages;
- BackOffice всё равно должен соблюдать единые design tokens, иконки и терминологию проекта.

## 6. Touch-first requirements for POS Front

POS Front проектируется прежде всего под touch/touchpad.

Обязательные правила:

- минимальная зона нажатия не меньше 48×48 px;
- основные POS-кнопки должны быть крупнее минимального значения;
- важные действия должны быть визуально широкими;
- иконки POS должны быть крупнее backoffice-иконок;
- scroll controls должны быть доступны пальцем;
- нельзя полагаться на hover как на основной способ раскрытия информации;
- long press допустим только как дополнительный путь, но не как единственный путь к важной операции;
- фокус, selected state, disabled state и locked state должны быть хорошо различимы.

## 7. Внутренние стандарты проекта

Основной baseline проекта:

1. **POS/RMS Operational UI Rules** — внутренние правила для кассы, заказов, столов, смен, оплат, возвратов и модальных потоков.
2. **ISO 9241 principles** — эргономика взаимодействия человек-система, пригодность для задачи, предсказуемость, устойчивость к ошибкам.
3. **Shneiderman’s Eight Golden Rules** — практические правила поведения интерфейса.
4. **Nielsen usability heuristics** — чеклист для UI review.
5. **Android/tablet usability considerations** — только как дополнительные рекомендации для touch, viewport, читаемости и размеров элементов.

WCAG не является основным дизайн-стандартом проекта. Его можно использовать как справочный чеклист для отдельных accessibility-проверок: контраст, фокус, labels, читаемые ошибки, modal focus management и совместимость со вспомогательными технологиями.

## 8. Правила Шнейдермана в контексте проекта

### 8.1 Согласованность

Одинаковые действия, иконки, цвета, модалки и состояния должны вести себя одинаково во всех UI.

### 8.2 Быстрые пути для опытных пользователей

POS должен поддерживать быстрые повторяемые операции без лишней навигации.

### 8.3 Информативная обратная связь

Каждое действие должно давать понятный результат: loading, success, warning, error, locked, disabled reason.

### 8.4 Завершенные диалоги

Каждый flow должен иметь начало, результат и безопасный выход.

Пример:

```text
Заказ -> Пречек -> Locked state -> Оплата/Отмена -> Результат
```

### 8.5 Предотвращение ошибок

UI должен предотвращать опасные действия до выполнения операции.

### 8.6 Возможность отмены

UI cancel и business cancellation должны быть различимы. Отмена окна не равна отмене финансовой операции.

### 8.7 Контроль пользователя

Система не должна неожиданно уводить пользователя из текущего контекста.

### 8.8 Снижение нагрузки на память

Пользователь должен видеть текущий контекст на экране, а не помнить его.
