# POS UX Guidelines

Статус: обязательные UX-правила для проектирования POS UI, дизайн-системы и будущих экранов POS/BackOffice.

Этот документ дополняет `design.md`. Если UX-идея конфликтует с фактическим backend/API/runtime, источником истины остаются код, тесты и backend specs. UI не должен показывать неподдержанную backend-функцию как рабочую.

## Цель

POS UI проектируется как рабочий инструмент ресторана в стрессовой среде: быстро выбрать стол, добавить блюда, выпустить пречек, принять оплату, обработать возврат/перепечатку и сверить кассу. Красота вторична относительно скорости, ясности, предотвращения ошибок и контроля состояния.

## Правила Шнейдермана как обязательная UX-база

Весь POS UI должен руководствоваться восемью золотыми правилами интерфейсного дизайна Бена Шнейдермана.

### 1. Стремиться к согласованности

- Одинаковые действия должны выглядеть и работать одинаково во всех разделах.
- `Пречек`, `Касса`, `Отмена`, `Void`, `Refund`, `Reprint`, `Retry`, `Manager override` не должны менять визуальную семантику от экрана к экрану.
- Иконки, цвета, статусы, отступы, размеры touch-targets, modal patterns и empty/error states должны быть едиными.

### 2. Давать опытным пользователям ускорители

- Частые действия должны быть доступны за 1-2 касания.
- Допустимые ускорители: favorites, quick groups, context chips, bottom quick access bar, long press, overflow, hotkeys для desktop при наличии touch-альтернативы.
- Ускоритель не может быть единственным способом выполнить действие.
- Для touch-устройств long press всегда дублируется action rail/overflow-путем.

### 3. Давать информативную обратную связь

- Каждая операция должна показывать состояние: выполняется, выполнено, ожидает синхронизации, ошибка, требуется действие.
- Пользователь должен понимать, что произошло с заказом, пречеком, оплатой, возвратом, чеком, сменой и кассовой операцией.
- Offline/sync индикаторы должны быть видимы, но не блокировать order entry, если backend/runtime разрешает локальную операцию.

### 4. Завершать диалоги понятным итогом

- Payment/refund/precheck cancel/manager override/cash drawer flow должен завершаться явным результатом.
- После закрытия modal пользователь должен понимать: заказ оплачен или нет, чек создан или нет, возврат выполнен или нет, операция отменена или требует действия.
- Нельзя оставлять пользователя в состоянии неопределенности после финансового действия.

### 5. Предотвращать ошибки

- Опасные, финансовые и необратимые действия требуют подтверждения и, при необходимости, manager override/RBAC.
- Disabled action должен объяснять причину недоступности, если причина не очевидна.
- UI не должен показывать функции со статусом `запланировано далее` или backend-unsupported функции как активные кнопки.
- Нельзя показывать raw SQL/Go errors, PIN, токены, request dumps и payment-sensitive payloads.

### 6. Разрешать отмену и исправление там, где это допустимо

- Где бизнес-правила позволяют, должны быть понятные flows: void, cancel, refund, reprint, retry.
- Если действие необратимо, юридически или фискально ограничено, UI обязан заранее объяснить последствия.
- Возврат/отмена не должны выглядеть как простое удаление, если операция уже имеет фискальные, платежные или ledger-последствия.

### 7. Сохранять ощущение контроля у пользователя

- Modal не должен менять текущий раздел и не должен терять выбранный стол/заказ.
- UI не должен неожиданно переводить пользователя из order entry в другой раздел без явного действия.
- Backend остается авторитетным, но UI обязан объяснять текущее состояние и следующий возможный шаг.

### 8. Снижать нагрузку на кратковременную память

- Текущий стол, заказ, гость/seat, статус пречека, сумма, остаток оплаты, смена, кассовая смена и sync state должны быть видимы или быстро доступны.
- Пользователь не должен помнить UUID, технические id, состояние оплаты, наличие issued precheck или количество outbox-событий.
- Short id допустим для операционного UI; внутренние идентификаторы показываются только в диагностике/debug context.

## Светлая и темная темы

POS UI обязан поддерживать две полноценные темы:

- светлую;
- темную.

Это обязательное архитектурное требование, а не косметическая настройка.

### Когда нужна светлая тема

- дневные рестораны и кафе;
- яркие помещения;
- backoffice-like сценарии;
- экраны с высокой информационной плотностью;
- администрирование, отчеты и настройка.

### Когда нужна темная тема

- бары;
- вечерние и ночные смены;
- затемненные залы;
- kitchen-adjacent зоны;
- терминалы, на которые долго смотрят при низком освещении.

### Общие правила тем

- Значение цвета не меняется между темами.
- Красный остается danger/destructive/error.
- Зеленый остается success/paid/ready, если это подтверждено контекстом.
- Желтый/оранжевый остается warning/pending/attention.
- Синий/нейтральный info не должен конкурировать с primary action.
- Цвет не является единственным носителем смысла: статус дополняется label, icon, badge, shape или helper text.
- Контраст текста, кнопок, disabled states и warning/error states должен проверяться отдельно для обеих тем.

## Design tokens

Компоненты не должны использовать raw/hardcoded colors. Все цвета должны идти через semantic tokens.

Минимальный набор POS-токенов:

```css
--pos-bg
--pos-surface
--pos-surface-raised
--pos-border
--pos-text-primary
--pos-text-secondary
--pos-text-disabled
--pos-action-primary
--pos-action-secondary
--pos-status-success
--pos-status-warning
--pos-status-danger
--pos-status-info
--pos-sync-pending
--pos-sync-error
--pos-payment-fiscal
--pos-payment-non-fiscal
--pos-selected-table
--pos-selected-order
```

Правило: компонент использует semantic token, а тема определяет фактический цвет. Компонент не должен знать, светлая сейчас тема или темная.

## Проверка critical flows в обеих темах

Каждый critical POS flow должен быть проверен в светлой и темной теме:

- PIN login;
- выбор зала/стола;
- ввод заказа;
- добавление позиции;
- выбор модификаторов;
- void строки;
- выпуск пречека;
- locked order после пречека;
- оплата;
- возврат/refund;
- перепечатка/reprint;
- cash in/cash out/no sale/cash count;
- offline/sync warning;
- stop-list conflict;
- loading state;
- empty state;
- error state;
- no-permission state.

## Критерий приемки

Новый экран или компонент нельзя считать готовым, если он:

- нарушает правила Шнейдермана;
- работает только в одной теме;
- содержит hardcoded colors вместо semantic tokens;
- показывает backend-неподдержанную функцию как активную;
- не имеет понятных loading/empty/error/no-permission states;
- требует hover-only или keyboard-only interaction для critical flow;
- оставляет пользователя в неопределенности после финансовой или destructive операции.
