# ADR-018: Внешние hardware-адаптеры (exe-плагины) для низкоуровневой работы с устройствами

Статус: принято.

Дата: 2026-07-01.

Ссылки: ADR-017 (шаблоны и печать), `docs/backend/RECEIPT-PRINT-SPEC.md`,
`docs/backend/EDGE-PRINT-ROUTING-SPEC.md`, `docs/backend/EDGE-HARDWARE-ADAPTER-PROTOCOL.md`,
POS-82…86, физическая проверка POS-86 на реальном USB-принтере 2026-07-01.

## Контекст

Физическая проверка POS-86 (2026-07-01, Xprinter XP-365B, Windows 11) показала два факта:

1. Обнаружение и адресация физических принтеров на Windows — низкоуровневая,
   версионно-хрупкая ОС-специфика: `\\.\USB001` (легаси-путь, на который до сих пор
   ссылалась документация) не открывается на современной Windows; реально рабочий путь —
   SetupAPI device interface (`\\?\USB#VID_...&PID_...#<serial>#{GUID_DEVINTERFACE_USBPRINT}`),
   который нужно получать динамически через PnP-перечисление конкретной машины.
2. Cloud API уже сознательно **не принимает** `address`/`port` для `type=usb`
   (`docs/backend/CLOUD-BACKEND-SPEC.md`, POS-82) — то есть архитектура уже была рассчитана
   на то, что физический адрес USB-принтера не может быть Cloud-owned значением. Но
   механизма, которым Edge узнаёт и сам управляет этим физическим адресом, до сих пор нет:
   ни обнаружения устройств, ни Edge-side API/UI, ни сканирования сети для `type=tcp`.

Отдельно: `pos-backend` — кроссплатформенный Go-модуль (задел на будущее вне Windows), а
низкоуровневая работа с принтерами (`os.OpenFile`/`CreateFile` по device path, Windows
print spooler/driver/queue API, PnP-перечисление, сетевой port-scan) является чисто
Windows-специфичной и, по сути, отдельной эксплуатационной обязанностью: она требует
других прав/окружения (иногда — административных операций типа `Add-Printer`,
`Add-PrinterDriver`, создания print queue), меняется с каждой версией Windows и не должна
тянуть за собой рост поверхности основного `pos-edge.exe`.

## Рассмотренные варианты

### A. Оставить всё в `pos-edge.exe`

Расширять существующий `shared/platform/receipt/escpos` пакет обнаружением/сканированием
прямо в основном процессе.

- ✅ Не нужен новый межпроцессный протокол.
- ❌ Тянет `golang.org/x/sys/windows`, PnP/WinAPI-специфику и spooler-код прямо в
  бизнес-процесс, который должен оставаться максимально простым и переносимым.
- ❌ Падение/зависание низкоуровневого Windows-вызова (например, зависший `CreateFile`
  на отвалившемся USB-порту) может уронить или подвесить весь Edge-процесс, включая
  кассовый флоу, не связанный с печатью.
- ❌ Обновление логики работы с конкретной моделью принтера требует пересборки и
  переустановки всего `pos-edge.exe`.

### B. Внешний exe-адаптер с процессной изоляцией и generic-протоколом (принято)

Низкоуровневая, ОС/hardware-специфичная логика выносится в отдельный исполняемый файл
("адаптер"), который Edge запускает и супервизирует как child-процесс и с которым
общается по локальному, транспортно-независимому protocol channel. Первый адаптер —
`windows-printers`: USB-обнаружение, сетевое сканирование, создание/выбор Windows print
queue, отправка уже отрендеренных ESC/POS-байт на физическое устройство.

- ✅ Крах/зависание адаптера не роняет `pos-edge.exe`: print worker получает контролируемую
  ошибку по таймауту и переводит job в обычный retry/failed путь без изменения финансового
  состояния.
- ✅ Windows-специфичный код (PnP, spooler, WinAPI) физически отделён от кроссплатформенного
  бизнес-кода; будущий адаптер под другую ОС/другое hardware (фискальный регистратор,
  весы, эквайринг) — это новый exe с тем же протоколом, без изменения core.
  Название `windows-printers` уже отражает это: `<платформа>-<домен>`.
- ✅ Обновление адаптера (например, добавить модель принтера или починить discovery)
  не требует пересборки/переустановки основного Edge exe.
- ❌ Появляется новый межпроцессный протокол, который нужно спроектировать один раз и
  правильно (см. `EDGE-HARDWARE-ADAPTER-PROTOCOL.md`), иначе получим N несовместимых
  ad-hoc протоколов под N будущих адаптеров.
- ❌ Дополнительная эксплуатационная сущность: адаптер нужно ставить, обновлять, следить
  за его живостью на каждой Edge-машине.

## Решение

Принят вариант B. `pos-edge.exe` остаётся единственным владельцем бизнес-логики
(routing, print_jobs, retry, RBAC, sync). Вся ОС-специфичная работа с физическими
устройствами инкапсулируется в отдельные exe-адаптеры, которые Edge запускает как
child-процессы и с которыми говорит по единому, адаптеро-независимому протоколу
(транспорт, кадрирование сообщений, discovery/print/health контракты — см.
`docs/backend/EDGE-HARDWARE-ADAPTER-PROTOCOL.md`). Первая реализация — `windows-printers`.

Ключевое архитектурное следствие: **логическая конфигурация принтера (Cloud-owned: имя,
CPL, codepage, тип отреза, набор document_types) и физическая привязка (Edge-owned:
конкретное USB-устройство/очередь или host:port) — это два разных объекта**. Cloud
никогда не является источником истины для физического адреса; Edge выбирает и хранит
физическую привязку локально, используя обнаружение, выполненное адаптером. Это прямое
продолжение уже принятого в POS-82 решения "USB принтеры не принимают address/port от
Cloud" — этот ADR формализует, как именно Edge закрывает эту часть.

## Последствия

- Новый компонент репозитория: `windows-printers/` (отдельный Go-модуль, отдельный exe,
  собирается и подписывается независимо от `pos-edge.exe`; входит в Windows-installer
  как отдельный установленный/автозапускаемый компонент).
- `pos-edge.exe` получает adapter host/supervisor: запуск, health-check, restart-with-backoff,
  graceful shutdown дочернего процесса, и клиент протокола (запрос discovery, отправка
  print-заданий, получение событий).
- Новая Edge-local таблица физической привязки принтера (см. protocol-doc) и новая
  Edge HTTP API поверхность для управления ей (RBAC `pos.print_routing.manage`, тот же
  permission, что уже владеет `print_routes`).
- Новый тип Edge -> Cloud external-события (обнаруженные устройства) и его обработка на
  Cloud стороне как *ephemeral candidate list*, а не master data.
- Существующий print worker (`pos-backend/internal/pos/app/print`) должен резолвить
  физический адрес не из `receipt_printers.address` напрямую, а через локальную
  физическую привязку с fallback на legacy `address`, если привязки ещё нет (переходный
  период). Это меняет `printerConfig()`/`Open()` call path — отдельная codegen-итерация,
  не входит в этот ADR как code change, только как обязательство будущей реализации.
- Не отменяет и не меняет уже принятое ADR-017 (шаблоны, IR, ESC/POS/SVG рендер) —
  адаптер получает уже отрендеренные байты, рендеринг остаётся в `pos-edge.exe`/
  `shared/platform/receipt`.

## Вне scope этого ADR

- Сам протокол (транспорт, формат сообщений, схемы событий) — вынесен в отдельный
  living-документ `docs/backend/EDGE-HARDWARE-ADAPTER-PROTOCOL.md`, чтобы его можно было
  развивать без правки ADR.
- Конкретная Cloud UI-форма выбора обнаруженного принтера — продуктовое решение,
  формализуется в `docs/ui/` в соответствующей codegen-итерации.
- Адаптеры для не-Windows платформ, фискальных регистраторов, весов, эквайринга —
  будущие адаптеры по тому же протоколу, вне текущего объёма.
- Удалённые/сетевые адаптеры (адаптер не на той же машине, что Edge) — вне текущего
  объёма; протокол проектируется transport-independent, но first-class сценарий — local
  loopback/named pipe на одной машине.
